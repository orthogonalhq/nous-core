# AGENTS

This repo follows the Nous-OSS Engineering Workflow SOP.
Source of truth is the repo docs. Manual approval at every gate.

## Read first
- docs/content/docs/sop/agent-engineering-workflow.mdx

## Core rules
1. Repo docs are the system of record
2. Work only moves forward when the required artifact exists
3. Nothing merges without verification
4. Nothing is complete until docs reflect reality
5. One sub-phase = one branch = one PR
6. Conventional commits required
7. Manual approval at every gate

## Three documents per sub-phase (no exceptions)
Every sub-phase produces:
1. **Goals** — why this work exists, scope, success criteria
2. **SDS** — design rationale, invariants, ADR review
3. **Implementation Plan** — explicit coding blueprint, file-by-file, todo list

Use templates at docs/content/docs/worklog/templates/. Fill in ALL sections. N/A with justification if not applicable. Never delete sections.

## Workflow gates (stop if missing)

1) Goals
- Goals doc exists: worklog/phase-X/phase-X.Y/goals.mdx
- Scope and success criteria included
- Links to relevant architecture docs
- Approved by human

2) Design
- SDS exists: worklog/phase-X/phase-X.Y/sds.mdx
- ADR Review section completed (existing ADRs checked, new ADRs identified or explicitly none)
- ADR created if durable decision: worklog/adr/NNN-title.mdx
- Approved by human

3) Plan
- Implementation Plan exists: worklog/phase-X/phase-X.Y/implementation-plan.mdx
- Includes explicit task list (file by file, function by function), todo checklist, tests with descriptions, acceptance criteria, rollback plan
- PR Compliance Checklist completed
- Approved by human

4) Implement + Verify
- Branch: phase-X.Y/short-description
- PR includes links to all three worklog artifacts
- CI green (lint/typecheck/tests)
- Acceptance criteria met
- Docs updated in same PR if reality changed
- Approved by human

5) Ratify
- Reference docs updated to match reality
- ADR updated with PR link (if ADR exists)
- SDS updated if implementation diverged

## ADR rules
- Write an ADR when the decision is durable, cross-cutting, hard to reverse, or changes contracts/ops posture
- BEFORE making any significant architectural decision, review ALL existing ADRs to avoid contradictions
- ADRs are numbered sequentially: 001-title.mdx, 002-title.mdx, etc.
- ADRs are never deleted. Superseded ADRs link to their replacement.
- Do not contradict a previous ADR without creating a new ADR that explicitly supersedes it.

## Testing philosophy
- Tests ship with every PR. No exceptions.
- Tier 1: Contract tests (critical) — does the implementation fulfill the interface?
- Tier 2: Behavior tests (important) — does the system do what the user expects?
- Tier 3: Edge case/regression tests (as needed)
- Target: 80% coverage on behavioral code. If below, explain why in the PR.
- Do NOT write tests that only exist to pad coverage.

## Change size guide
- Small (bug fix, config, minor refactor): Brief Goals/SDS/Plan. Tests if behavioral.
- Medium (new package, new interfaces): Full Goals/SDS/Plan. Tests required.
- Large (new layer, cross-cutting change): Comprehensive Goals/SDS/Plan. ADR(s) likely. Tests required.

## Branch naming
Format: phase-X.Y/short-description
Example: phase-1.1/shared-types-interfaces-config

## Conventional commits
Format: type(scope): description
Types: feat, fix, test, docs, refactor, chore
Scope: package or layer name (e.g., shared, autonomic/storage, cortex/pfc)
