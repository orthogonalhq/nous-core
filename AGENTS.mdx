# AGENTS

This repo follows the Nous-OSS Engineering Workflow SOP.
Source of truth is the repo docs. Manual approval at every gate.

## Read first
- docs/content/docs/sop/agent-engineering-workflow.mdx

## Dispatch model
Work uses three roles: PFC (project owner) dispatches, agents execute, fresh agents review.
- **Agent A (prompt generator):** reads docs, produces phase prompt, done (disposable).
- **Agent B (implementer):** receives approved prompt, executes full SOP cycle (Goals → SDS → Impl Plan → Code → Completion Report). Fresh per sub-phase. Same agent for revisions within a sub-phase.
- **Review agents (gate reviewers):** fresh disposable agents that review each artifact at each gate. One per gate review. PFC incorporates their output into the final Review document.
- **PFC:** dispatches all agents, approves or rejects at every gate, writes the final Review document. Never implements.

## Prompt templates
- Phase prompt: docs/content/docs/worklog/templates/phase-prompt.mdx — generates the prompt for Agent B
- Gate review prompt: docs/content/docs/worklog/templates/gate-review-prompt.mdx — generates review prompts for each gate (Goals, SDS, Implementation Plan, Completion Report)

See SOP "Dispatch Model" section for full details.

## Core rules
1. Repo docs are the system of record
2. Work only moves forward when the required artifact exists
3. Nothing merges without verification
4. Nothing is complete until docs reflect reality
5. One sub-phase = one branch = one PR
6. Conventional commits required
7. Manual approval at every gate

## Five documents per sub-phase (no exceptions)
Every sub-phase produces:
1. **Goals** — why this work exists, scope, success criteria (before work, by implementer)
2. **SDS** — design rationale, invariants, ADR review (before work, by implementer)
3. **Implementation Plan** — explicit coding blueprint, file-by-file, todo list (before work, by implementer)
4. **Completion Report** — what was built, verification results, deviations, acceptance criteria (after work, by implementer)
5. **Review** — SOP compliance check, issues found, actions taken, verdict (after work, by reviewer/PFC)

Use templates at docs/content/docs/worklog/templates/. Fill in ALL sections. N/A with justification if not applicable. Never delete sections.

## Workflow gates (stop if missing)

1) Goals
- Goals doc exists: worklog/phase-X/phase-X.Y/goals.mdx
- Scope and success criteria included
- Links to relevant architecture docs
- Approved by human

2) Design
- SDS exists: worklog/phase-X/phase-X.Y/sds.mdx
- ADR Review section completed (existing ADRs checked, new ADRs identified or explicitly none)
- ADR created if durable decision: worklog/adr/NNN-title.mdx
- Approved by human

3) Plan
- Implementation Plan exists: worklog/phase-X/phase-X.Y/implementation-plan.mdx
- Includes explicit task list (file by file, function by function), todo checklist, tests with descriptions, acceptance criteria, rollback plan
- PR Compliance Checklist completed
- Approved by human

4) Implement + Verify
- Branch: phase-X.Y/short-description
- PR includes links to all three worklog artifacts
- CI green (lint/typecheck/tests)
- Acceptance criteria met
- Docs updated in same PR if reality changed
- Approved by human

5) Ratify
- Completion Report exists: worklog/phase-X/phase-X.Y/completion-report.mdx
- Status, commits, what was built, verification, deviations, acceptance criteria
- Reference docs updated to match reality
- ADR updated with PR link (if ADR exists)
- SDS updated if implementation diverged
- Tech stack doc updated if tooling deviations occurred

6) Review
- Review exists: worklog/phase-X/phase-X.Y/review.mdx
- Performed by reviewer/PFC, NOT the implementer
- SOP compliance check, issues found, actions taken, SOP improvements
- Verdict: Approved / Approved With Actions / Needs Revision
- If Needs Revision → back to implementer. Otherwise → sub-phase complete.

## ADR rules
Apply this logic test to every deviation, tooling change, or design decision:
1. Would a future agent need to know this before making a related decision? → ADR
2. Will this decision be repeated in future sub-phases? → ADR
3. Does this change contradict or modify what a reference doc says? → ADR + update reference doc
4. Is the decision durable, cross-cutting, hard to reverse, or changes contracts/ops posture? → ADR
If ANY are true, create an ADR before completing the sub-phase. Do not defer.
- BEFORE making any significant architectural decision, review ALL existing ADRs to avoid contradictions
- ADRs are numbered sequentially: 001-title.mdx, 002-title.mdx, etc.
- ADRs are never deleted. Superseded ADRs link to their replacement.
- Do not contradict a previous ADR without creating a new ADR that explicitly supersedes it.

## Testing philosophy
- Tests ship with every PR. No exceptions.
- Tier 1: Contract tests (critical) — does the implementation fulfill the interface?
- Tier 2: Behavior tests (important) — does the system do what the user expects?
- Tier 3: Edge case/regression tests (as needed)
- Target: 80% coverage on behavioral code. If below, explain why in the PR.
- Do NOT write tests that only exist to pad coverage.

## Change size guide
- Small (bug fix, config, minor refactor): Brief Goals/SDS/Plan. Tests if behavioral.
- Medium (new package, new interfaces): Full Goals/SDS/Plan. Tests required.
- Large (new layer, cross-cutting change): Comprehensive Goals/SDS/Plan. ADR(s) likely. Tests required.
