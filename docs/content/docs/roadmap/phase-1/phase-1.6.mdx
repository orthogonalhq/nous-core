# Phase 1.6 — Web UI

**Branch:** `phase-1.6/web-ui-chat-projects-traces`

**Size:** Large

**Contributes to:** Milestone 5 (Knows Project), Milestone 6 (See What's Happening), Milestone 7 (Looks Good)

**Prerequisite:** Phase 1.5 (the full execution loop must be working)

---

## Goals

Build the primary interaction surface. When 1.6 is done, a user can open a browser, see their projects in a sidebar, have a conversation with Nous, view execution traces, inspect memory, and modify configuration — all through a clean, modern UI.

---

## Scope

### In Scope

- `self/apps/web/` — Next.js application (App Router):

  **Application Shell:**
  - Nous is the shell — navigation, layout, design system, routing
  - Built with Next.js App Router for file-based routing and layouts
  - shadcn/ui + Tailwind CSS for the component library and styling
  - The shell is designed from day one to support future package UI surfaces (federated micro frontend model — packages render within the shell in later phases)
  - Shared layout: persistent sidebar + content area

  **Chat Interface:**
  - Conversational interaction with Nous
  - Streamed response display (token by token via tRPC subscription)
  - Message history (loaded from STM)
  - Input field with send action

  **Project Sidebar:**
  - List all projects
  - Create new project
  - Switch between projects (conversation context switches)
  - Active project indicator
  - Project-scoped conversation history

  **Trace Display:**
  - View execution traces for the current project
  - Each trace shows: PFC decisions, model called, tool authorizations, memory writes/denials, timing
  - Expandable detail for each trace step

  **Memory Inspector:**
  - View stored memories for the current project
  - See what was approved and what was denied
  - Memory export action
  - Memory delete action

  **Configuration Panel:**
  - View current model role assignments
  - View/modify PFC tier
  - View/modify project-specific settings
  - System health: is the local model running, is storage healthy

  **Design:**
  - Responsive layout (desktop and tablet)
  - Dark mode support
  - Clean, modern aesthetic

- API layer (tRPC):
  - tRPC router connecting the web frontend to the cortex — end-to-end type-safe
  - Subscriptions over WebSocket for streaming (model responses, trace updates)
  - Queries and mutations over HTTP for CRUD (projects, memory, config)
  - Localhost-only by default
  - Same tRPC router consumed by the CLI (Phase 1.8)

### Out of Scope

- Workflow editor / node-based visual editor (Phase 5)
- Dashboard with project metrics (Phase 3)
- Messaging bridge (Phase 9)
- Mobile-native app (Phase 9)
- Advanced search or filtering of memories (Phase 2)

## Architecture Notes

The web UI is built as a **shell** from day one. In Phase 1, only Nous's own pages render inside the shell. In Phase 7+, installed packages can declare menu items, pages, and tabs in their manifest — the shell renders them using a federated micro frontend model. Packages build to the SDK. Nous orchestrates the rendering.

This means Phase 1.6 must establish:
- A navigation system that can be extended dynamically (not hardcoded routes)
- A layout system with slots where future package content can render
- A tRPC client pattern that packages can follow

These don't need to be fully implemented for packages in Phase 1, but the architecture must not prevent them.
