# Phase 1 — Core Runtime

Phase 1 establishes the control plane of Jarvis.

This phase defines how models are orchestrated, how memory is persisted, how tools are executed, and how projects are isolated. Every future phase depends on the stability of these decisions.

Jarvis is not being built as a chatbot wrapper.  
It is being built as a governed agent runtime.

---

## Objective

Deliver a beta-ready local-first agent runtime that is:

- Architecturally deterministic  
- Role-based in model routing  
- Explicit in memory persistence  
- Strict in tool authorization  
- Cross-platform compatible  
- Configurable across hardware tiers  

At the end of Phase 1, a user should be able to install Jarvis, configure a PFC tier, select local or remote models, create projects, persist memory, and execute governed tool workflows.

---

## Architectural Spine

Phase 1 is organized around five core pillars.

### 1. PFC Orchestration Model

The Prefrontal Cortex is the control plane.

It is responsible for:

- Routing model roles  
- Performing self-reflection  
- Approving memory writes  
- Authorizing tool execution  
- Enforcing policy boundaries  

Only the PFC may execute tools or persist memory.

PFC Modes:

0 — Off  
1 — Mobile  
2 — Low spec PC  
3 — Mid spec PC  
4 — High spec PC  
5 — Remote agent  

Each tier adjusts reasoning depth and orchestration strength.

Local PFC execution targets a WASM sandbox where feasible.

---

### 2. Role-Based Model System

Jarvis does not treat “the LLM” as a monolith.

Models are assigned roles:

- Orchestrator (PFC)  
- Reasoner  
- Tool advisor  
- Summarizer  
- Embedder  
- Reranker  
- Vision  

Each role may be fulfilled by a local or remote provider.

Specialist models advise the PFC.  
They do not execute tools or write memory directly.

---

### 3. Explicit Memory Architecture

Memory is structured and intentional.

Two layers exist:

Short Term Memory  
- Recent context  
- Working state  
- Summaries  

Long Term Memory  
- Structured facts and preferences  
- Project-scoped knowledge  
- Vector-backed semantic store  

All memory writes originate as `MemoryWriteCandidate` objects and require PFC approval before persistence.

Memory is:

- Project-scoped  
- Exportable  
- Deletable  
- Auditable  

---

### 4. Project Isolation

Projects are first-class containers.

Each project encapsulates:

- Model configuration  
- Memory namespace  
- RAG index  
- Tool permissions  
- Execution traces  

No implicit cross-project leakage is allowed.

---

### 5. Tool Execution Model

Tools are capability-gated and schema-defined.

Each tool:

- Declares permissions  
- Declares required capabilities  
- Defines typed input and output  
- Executes within controlled boundaries  

Tools never receive direct ambient authority to the system.

All execution is mediated by the PFC.

---

## Local and Remote Support

Phase 1 supports multiple execution profiles:

- Local PFC + Remote specialists  
- Local PFC + Local specialists  
- Remote PFC + Remote specialists  
- PFC Off  

Model providers are abstracted through normalized adapters.

Remote models never receive unfiltered system state.

---

## Cross-Platform Runtime

Jarvis must support:

- macOS  
- Linux  
- Windows (including WSL2)  

OS-specific logic is isolated behind system interfaces.

Local model execution should support native runtime and WASM where feasible.

---

## Beta Exit Criteria

Phase 1 reaches beta when the system can:

- Install and run locally  
- Configure PFC tier  
- Configure local and/or remote models  
- Create and switch projects  
- Ingest documents for retrieval  
- Persist STM and LTM across sessions  
- Execute capability-gated tools  
- Stream responses  
- Inspect execution traces  
- Export or delete memory  

No marketplace functionality is included in Phase 1.

---

Phase 1 defines the architectural contract of Jarvis-OSS.

Everything that follows depends on its stability.
