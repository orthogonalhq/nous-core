# Phase 1.1 — Core Interfaces and Runtime Spine

Phase 1.1 defines the structural backbone of Jarvis.

Before optimizing models, memory depth, or tool ecosystems, the runtime must be deterministic and interface-driven.

This phase establishes the minimal stable contracts that every other component depends on.

---

## Objective

Define and implement the core interfaces that:

- Separate orchestration from execution
- Abstract model providers
- Isolate tool execution
- Encapsulate storage
- Enable deterministic routing

This phase creates the architectural skeleton of Jarvis.

---

## Runtime Flow (High Level)

A single agent cycle follows this structure:

1. Input received
2. Context constructed (STM + project + retrieval)
3. PFC evaluates
4. Optional specialist model consulted
5. PFC validates output
6. Tool execution authorized if required
7. Memory write candidates generated
8. PFC approves/denies memory writes
9. Response streamed
10. Trace persisted

Every step is explicit.

---

## Core Interfaces

### ModelProvider

Abstracts any model backend.

Responsibilities:
- Accept structured prompt input
- Return structured response
- Support streaming
- Provide token usage metadata
- Normalize provider-specific formats

Providers may be:
- Local runtime
- WASM model
- Remote API

---

### ModelRouter

Maps model roles to providers.

Responsibilities:
- Resolve role → provider
- Apply profile rules
- Handle fallback and failover
- Enforce privacy mode restrictions

Routing decisions are deterministic and logged.

---

### PfcEngine

The orchestration control plane.

Responsibilities:
- Reflection
- Escalation decisions
- Tool authorization
- Memory write gating
- Policy enforcement

Only the PFC may authorize side effects.

---

### Tool Interface

Defines executable capabilities.

Each tool must declare:

- Name
- Version
- Input schema
- Output schema
- Required capabilities
- Permission scope

Tools execute only through the PFC.

---

### MemoryStore

Abstracts STM and LTM persistence.

Responsibilities:
- Store short-term working state
- Persist long-term structured memory
- Persist semantic embeddings
- Track provenance
- Support deletion and export

Memory writes must be explicit events.

---

### ProjectStore

Encapsulates project isolation.

Responsibilities:
- Namespace memory
- Store model configuration
- Maintain tool permission sets
- Persist execution traces

Projects are first-class boundaries.

---

## Non-Negotiable Rule

Only the PFC may:

- Execute tools
- Persist memory
- Modify project configuration

Specialist models never produce side effects directly.

---

Phase 1.1 ends when these interfaces are implemented and testable with stub providers.
