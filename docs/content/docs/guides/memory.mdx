# Memory

Nous stores approved memory entries per project. The Memory inspector lets you review memory state, mutation history, and deletion proofs.

## Approved Entries

Approved entries are persisted after the Cortex evaluates memory write candidates. Each entry includes:

- Content
- Type (fact, preference, experience-record, etc.)
- Scope (global, project)
- Provenance (trace, source, timestamp)
- Mutability class and lifecycle status
- Supersession linkage (`supersededBy`) when an entry is replaced

### Experience Records

Experience records are outcome-labeled memories with sentiment. They form the foundation of Nous's learning system. Each experience record has a canonical structure:

- **Context** â€” the situation as it was understood at decision time
- **Action** â€” what was done or proposed
- **Outcome** â€” what happened (approved, rejected, succeeded, failed, etc.)
- **Reason** â€” why the outcome occurred (user-stated or Cortex-inferred)
- **Sentiment** â€” valence of the outcome (see Sentiment Scale below)

All experience-record writes are policy-evaluated. Cross-project writes require explicit policy approval and are trace-linked for audit.

### Sentiment Scale

Experience records use a canonical sentiment scale for retrieval and confidence consumers:

| Sentiment | Meaning |
|-----------|---------|
| `strong-positive` | User approved immediately; high-confidence positive outcome |
| `weak-positive` | User approved with caveats or after adjustment |
| `neutral` | No clear valence; informational or mixed outcome |
| `weak-negative` | User rejected or outcome was negative with mitigating factors |
| `strong-negative` | User rejected firmly; high-confidence negative outcome |

Strong signals (positive or negative) weight higher than weak signals in retrieval scoring.

### Distilled Patterns

Experience records are periodically compressed into **distilled patterns** â€” consolidated knowledge derived from clusters of related experiences. This is the "sleep cycle" â€” consolidation that happens offline, not at query time.

Each distilled pattern includes:

- **Content** â€” synthesized from source records (context â†’ outcome: reason)
- **Confidence** â€” assigned from sentiment consistency and volume; refreshed on confirming signals, decayed on staleness or contradiction
- **Provenance** â€” `basedOn` and `supersedes` link to source experience records; `evidenceRefs` link to clustering and write evidence
- **Scope** â€” project-scoped by default; cross-project distillation requires policy evaluation before write

**Confidence lifecycle:** Patterns gain confidence when new experience records align (refresh); they lose confidence when contradicted or when they become stale (decay). Contradiction below a threshold flags the pattern for retirement.

**Supersession reversal:** If a distilled pattern is found to be incorrect, `reverseSupersession` restores the source records to active and retires the pattern. The caller must emit an audit record for the reversal (ADR-002 evidence-chain). No silent replacement.

### Retrieval and Policy

Retrieval is policy-enforced. All retrieval paths run through `PolicyEnforcedRetrievalEngine`, which evaluates access policy before delegating to the inner retrieval engine. When policy denies a retrieval (for example, global scope with `inheritsGlobal: false`), the response includes `policyDenial` â€” a `PolicyDecisionRecord` with reason code, trace ID, and evidence refs for audit.

Retrieval responses have the shape `{ results, policyDenial? }`. Use `response.results` for the scored entries; use `response.policyDenial` when present to trace denied or degraded outcomes.

## Lifecycle and Mutations

Memory entries move through governed lifecycle states:

- `active` â€” current durable value
- `superseded` â€” replaced by a newer entry (linked by `supersededBy`)
- `soft-deleted` â€” hidden from active use, but retained for audit/recovery
- `hard-deleted` â€” payload redacted with immutable tombstone proof retained

Governed mutation procedures are exposed through the API:

- `memory.supersede` to replace a value without in-place rewrite
- `memory.promote` and `memory.demote` to move entries between project/global placement states
- `memory.delete` for soft-delete (default) and hard-delete (with rationale)

## Denied Entries

When the Cortex denies a memory write or mutation, the candidate/action and reason are logged. Use:

- `memory.denials` for write denials
- `memory.audit` for mutation allow/deny/failed outcomes and reason codes

## Export and Audit Visibility

Export all memory for a project as structured data:

- `memory.export` -> `{ stm, entries, audit, tombstones }`

This gives a full operator view:

- Short-term conversation context (`stm`)
- Durable memory entries and lifecycle state (`entries`)
- Mutation history (`audit`)
- Hard-delete proof artifacts (`tombstones`)

## Delete Behavior

- **Soft-delete (default):** preserves entry content and audit history while marking lifecycle state.
- **Hard-delete:** requires `hard: true` plus `rationale`; payload content is replaced with `[hard-deleted]` and a tombstone record is created.
- **Delete all for project:** applies soft-delete to active project entries and clears STM context.
