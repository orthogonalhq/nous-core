# Public MCP Edge

Phase 13.2 enables the first public memory-operation floor for approved external clients. This surface is hosted by the web app, but it remains a thin adapter over the canonical gateway runtime and internal MCP catalog.

## Endpoints

| Endpoint | Method | Purpose |
|---|---|---|
| `/mcp` | `POST` | Streamable HTTP MCP transport with JSON-RPC framing for `initialize`, `tools/list`, and `tools/call` |
| `/.well-known/oauth-protected-resource/mcp` | `GET` | Protected Resource Metadata for the public MCP resource |
| `/.well-known/oauth-authorization-server/mcp` | `GET` | Authorization Server Metadata for the public MCP resource |

## Admission posture

- `/mcp` requires OAuth bearer posture before any tool mapping, namespace bootstrap, or shared handler execution.
- Admission validates request schema, bearer presence, token validity, audience, origin, and scopes at the edge.
- External callers are represented as `ExternalClient` subjects at the public boundary; the internal Worker, Orchestrator, Cortex:System, and Cortex:Principal classes are unchanged.
- Public tool names stay namespaced as `ortho.*` and are mapped once onto the canonical flat internal handler catalog.

## Phase 13.2 availability

`tools/list` returns only the intersection of caller scopes and current-phase-enabled public mappings.

### Enabled tools

| Public name | Purpose | Required scope |
|---|---|---|
| `ortho.memory.v1.put` | Append or supersede one source-local STM/LTM entry | `ortho.memory.stm.write` or `ortho.memory.ltm.write` |
| `ortho.memory.v1.get` | Fetch one source-local STM/LTM entry | `ortho.memory.stm.read` or `ortho.memory.ltm.read` |
| `ortho.memory.v1.search` | Search source-local STM/LTM entries | `ortho.memory.stm.read`, `ortho.memory.ltm.read`, or both |
| `ortho.memory.v1.delete` | Soft-delete one source-local STM/LTM entry | `ortho.memory.stm.delete` or `ortho.memory.ltm.delete` |
| `ortho.memory.v1.compact` | Run external-only `summarize` or `extract_facts` compaction | `ortho.memory.stm.write` for `summarize`, `ortho.memory.ltm.write` for `extract_facts` |

Future public tool names that are already reserved in the mapping table still fail closed with reason-coded `phase_not_enabled` responses until their owning phases land.

## Namespace and audit behavior

- Namespace identity is derived from validated client identity rather than caller-supplied storage coordinates.
- First-write bootstrap creates only `external:*` collections and records the manifest in `public_mcp_namespace_registry`.
- Durable admits and security-sensitive rejects emit witness-linked audit projections in `public_mcp_audit_projection`.
- Quota and rate-limit enforcement are keyed by the derived external namespace plus a server-generated token fingerprint. Clients do not send the fingerprint; the edge derives it from the validated bearer token and never stores the raw token.

## OAuth example

Use your OAuth authorization server's client-credentials flow to request the scopes you need. The discovery documents served from the `/.well-known/.../mcp` endpoints tell clients which authorization server and resource identifier to use.

```bash
curl -X POST https://auth.example.com/oauth/token \
  -H "content-type: application/x-www-form-urlencoded" \
  -u "client-123:client-secret" \
  --data-urlencode "grant_type=client_credentials" \
  --data-urlencode "audience=urn:nous:ortho:mcp" \
  --data-urlencode "scope=ortho.memory.stm.read ortho.memory.stm.write ortho.memory.stm.delete"
```

Example response:

```json
{
  "access_token": "eyJhbGciOi...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "ortho.memory.stm.read ortho.memory.stm.write ortho.memory.stm.delete"
}
```

### `ortho.memory.v1.compact`

`compact` is external-only and supports exactly two strategies in Phase 13.2:

- `summarize` writes a derived STM summary entry and requires `ortho.memory.stm.write`
- `extract_facts` writes derived LTM fact entries and requires `ortho.memory.ltm.write`

Example `summarize` request:

```json
{
  "jsonrpc": "2.0",
  "id": "rpc-compact-1",
  "method": "tools/call",
  "params": {
    "name": "ortho.memory.v1.compact",
    "arguments": {
      "namespace": "app:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
      "sourceTier": "stm",
      "strategy": "summarize",
      "maxEntries": 10,
      "idempotencyKey": "compact-summary-001"
    }
  }
}
```

```json
{
  "jsonrpc": "2.0",
  "id": "rpc-compact-1",
  "result": {
    "strategy": "summarize",
    "sourceTier": "stm",
    "sourceEntryCount": 4,
    "derivedEntryIds": ["cmp-1"]
  }
}
```

## Error behavior

Common public reject reasons in Phase 13.2:

| Reject reason | Meaning |
|---|---|
| `scope_insufficient` | The bearer token does not include the tier/operation-specific scope required by the request. |
| `namespace_unauthorized` | The request namespace does not match the namespace derived from the validated client identity. |
| `request_schema_invalid` | The JSON-RPC request or tool arguments failed schema validation. |
| `source_quarantined` | The source is quarantined or purged for durable mutation paths. |
| `quota_exceeded` | The request exceeds the namespace + token quota window. |
| `rate_limited` | The namespace + token has exceeded the allowed request rate for the tool. |
| `phase_not_enabled` | The tool is reserved for a later public phase. |

## Operator configuration

Phase 13.2 introduces no new public-memory-specific environment variables. The existing `NOUS_PUBLIC_BASE_URL` setting remains the only public MCP edge environment override and continues to control the base URL emitted in the OAuth discovery documents.

## Runtime boundary

The public MCP edge is not a parallel runtime or a public-only business-logic stack. The web host parses HTTP and discovery traffic, `IPublicMcpGatewayService` owns admission and audit posture, and `PublicMcpExecutionBridge` hands admitted calls into the same AgentGateway, Internal MCP, and backpressure path used elsewhere in the system.
